## Общие принципы

### 1. Кто первый — того и тапки

> **Первым опубликовавший контракт владеет его. Второй подстраивается под него.**

На самом деле неважно, каким именно будет контракт — главное, чтобы:
- его можно было реализовать;
- он выполнял своё назначение.

Первый устанавливает контракт без согласований, второй выполняет его.

### 2. Когда второй может отказаться

Второй может отказаться выполнять контракт только при наличии **веских архитектурных причин**. Критерии:

| Причина | Пример |
|---|---|
| Существенно выше сложность реализации | Контракт требует хранить дерево узлов, но внутренняя архитектура выдаёт только плоский список — отказ весомен. |
| Существенно выше нагрузка при выполнении | Плоский список с `parentId` против дерева узлов: каждый элемент требует рекурсивный обход при каждом вызове |
| Невозможность реализации совсем | Внутренняя архитектура не хранит разрыв между событиями и не может вернуть `threadId` |

Простое несогласие с форматом, именами полей или структурой вложения — **не являются** вескими архитектурными причинами.

### 3. Изменение первичного контракта

Если второй считает, что контракт нужно изменить:

```
1. Второй открывает Pull Request с изменениями в BRD/SRS и обоснованием
2. Второй ждёт ответа от создавшего контракт
3. Создавший: соглашается или отказывает с обоснованием
4. При отказе: второй подстраивается, сроки блокирующихся ошибок растут (exit 1/3)
5. При согласии: оба обновляют свои реализации
```

> Одностороннее изменение контракта без согласования владельца — нарушение регламента.

### 4. Оркестратор не ждёт исполнителя

Оркестратор реализует код по заявленному контракту независимо.  
Пока исполнитель не готов, оркестратор получает `exit 1` или `exit 3` — агент сориентируется по exit code и ждёт.  
Когда исполнитель выполняет контракт верно — оркестратор работает без правок.

### 5. Контракт должен знать своих потребителей

В каждом BRD/SRS должно быть указано, кто и как использует данный метод.  
Это позволяет оценить влияние любых изменений до того, как они будут внесены.

```markdown
## Потребители контракта

| Компонент | Роль | Как использует |
|---|---|---|
| `tracer-sink` CLI | Оркестратор | Шаг N сценария X, ждёт ответ до timeoutMs |
| Фильтрация выполняется на стороне CLI | Постобработка | Получает полный список, фильтрует сам по `variables[]` |
```

## Стандарт документации

Каждый метод описывается парой документов:

| Документ | Назначение | Пишет |
|---|---|---|
| `BRD-{method}.md` | Зачем: бизнес-проблема, цель, потребители | Тот, кто выдвигает требование |
| `SRS-{method}.md` | Как: интерфейс, контракт, ошибки | Тот, кто устанавливает контракт |

BRD и SRS должны содержать раздел **Потребители контракта** с указанием компонента, роли и способа использования.

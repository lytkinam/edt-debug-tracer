## Общие принципы

### 1. Кто первый — того и тапки

> **Первым опубликовавший контракт владеет им. Второй подстраивается под него.**

На самом деле неважно, каким точно будет контракт — главное, чтобы:
- его можно было реализовать;
- он выполнял своё назначение.

Первый устанавливает контракт без согласований, второй выполняет.

### 2. Когда второй может отказаться

Второй может отказаться выполнять контракт только при наличии **веских архитектурных причин**. Критерии:

| Причина | Пример |
|---|---|
| Существенно выше сложность реализации | Контракт требует хранить дерево узлов, но внутренняя архитектура выдаёт только плоский список — отказ весом |
| Существенно выше нагрузка при выполнении | Плоский список с `parentId` вместо дерева: каждый вызов требует рекурсивного обхода — отказ весом |
| Невозможность реализации | Внутренняя архитектура не хранит разрыв между событиями и не может вернуть `threadId` |

Простое несогласие с форматом, именами полей или структурой вложенности — **не является** веской архитектурной причиной.

### 3. Запрос на изменение первичного контракта

Если второй считает, что контракт нужно изменить:

```
1. Второй открывает Pull Request с изменениями BRD/SRS
2. PR должен содержать:
   а) обоснование необходимости изменений
   б) список изменений (что именно меняется)
   в) полное описание нового формата (не diff, а полный контракт)
3. Второй ждёт ответа от создавшего контракт
4. Создавший: соглашается или отказывает с обоснованием
5. При отказе: второй подстраивается, сроки блокирующих ошибок растут (exit 1/3)
6. При согласии: оба обновляют свои реализации
```

> **Особый случай — интерактивная разработка требований:**  
> Если разработка требований ведётся в интерактивном режиме с пользователем,  
> согласование может быть получено сразу при создании PR и зафиксировано в нём явно:  
> `Статус: Согласовано — [имя/роль], [дата]`

> Одностороннее изменение контракта без PR и согласования владельца — нарушение регламента.

### 4. Оркестратор не ждёт исполнителя

Оркестратор реализует код по заявленному контракту независимо.  
Пока исполнитель не готов — оркестратор получает `exit 1` или `exit 3`.  
Агент ориентируется по exit code и ждёт.  
Когда исполнитель выполняет контракт верно — оркестратор работает без правок.

### 5. Контракт знает своих потребителей

В каждом BRD/SRS должен быть раздел **«Потребители контракта»**:

```markdown
## Потребители контракта

| Компонент | Роль | Как использует |
|---|---|---|
| `tracer-sink` CLI | Оркестратор | Шаг N сценария X, ждёт ответ до timeoutMs |
```

Это обязательное поле. Без него невозможно оценить влияние любых изменений.

## Версионирование контрактов

Каждый BRD и SRS содержит поле `Версия` в заголовке.

| Тип изменения | Версия | Пример |
|---|---|---|
| Исправление опечатки, уточнение формулировки | patch: `1.0` → `1.0.1` | Уточнили описание поля |
| Добавление нового опционального поля | minor: `1.0` → `1.1` | Добавили `session_id` как опциональный |
| Изменение структуры, удаление поля, смена типа | major: `1.0` → `2.0` | `threadId` стал обязательным вместо опционального |

**Правила:**
- Major-изменение требует явного согласования всех потребителей (см. принцип 3).
- Minor и patch — через PR, согласование упрощённое.
- Версия контракта отражается в заголовке файла и в `docs/CHANGELOG.md`.

## Стандарт документации метода

Каждый метод описывается парой документов:

| Документ | Назначение | Пишет |
|---|---|---|
| `BRD-{method}.md` | Зачем: бизнес-проблема, цель, потребители | Тот, кто выдвигает требование |
| `SRS-{method}.md` | Как: интерфейс, контракт, ошибки | Тот, кто устанавливает контракт |

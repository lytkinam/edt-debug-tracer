# 07. Прямое подключение к HTTP debug endpoint 1С

## Штатный механизм

1C:Предприятие поддерживает режим **«отладка по протоколу HTTP»**. При его включении сервер или публикация поднимают debug endpoint, к которому подключается EDT или Конфигуратор.

Адрес формата: `https://<server>/<base>/debugger/`

## Как включить на сервере

В реестре службы сервера 1С (Windows):
```
HKLM\SYSTEM\CurrentControlSet\Services\1C:Enterprise 8.3 Server
ImagePath += " -debug -http"
```

В конфигураторе: `Сервис → Параметры → вкладка Отладка → Протокол: HTTP`.

## Ограничения прямого подключения

1C **не публикует спецификацию** низкоуровневого debug-протокола.
Это значит, что создать собственный клиент можно только через:
- **реверс-инжиниринг** трафика EDT/Конфигуратора (Wireshark/Fiddler),
- **внедрение в EDT** (наш подход — рекомендуемый).

## Сравнение двух подходов

| | EDT-плагин (наш подход) | Прямой HTTP клиент |
|--|------------------------|--------------------|
| Сложность | Средняя (Java + Eclipse PDE) | Высокая (реверс протокола) |
| Стабильность | Высокая (официальный API) | Низкая (протокол может меняться) |
| Совместимость | EDT 2024+ | Требует исследования под каждую версию |
| Независимость от IDE | Нет (нужен EDT) | Да |
| Рекомендуется | ✅ Да | ⚠️ Только для спецзадач |

## Вывод

Для задачи «трассировка шагов BSL + интеграция с Vanessa» **использовать EDT как клиент отладки и перехватывать события через Eclipse Debug API** — единственный надёжный подход без реверс-инжиниринга.

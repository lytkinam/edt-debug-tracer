# BRD — CLI-Сток: метод `breakpoint-metrics`

**Версия:** 1.0  
**Дата:** 2026-06-06  
**Репозиторий:** lytkinam/edt-debug-tracer, ветка `feature/trace-sink`  
**Статус:** Draft

---

## 1. Бизнес-контекст

CLI-сток — детерминированный оркестратор конкретных сценариев использования
EDT Debug Tracer. Каждый сценарий — отдельный метод с собственным контрактом.

Метод `trace-session` (см. BRD-trace-session.md) прогоняет полный цикл трейса
и возвращает все шаги подряд. Это полезно для навигации по коду, но
**не отвечает на вопрос**: «каково состояние программы в конкретной точке
останова?» — значения переменных, стек вызовов, контекст выполнения.

Для ИИ-агентов, выполняющих анализ поведения 1C-кода, критически важно
получить **срез состояния в заданной точке** без необходимости анализировать
весь трейс вручную.

---

## 2. Бизнес-проблема

| Проблема | Следствие |
|----------|-----------|
| `trace-session` возвращает все шаги, но не значения переменных в точке | Агент вынужден постобрабатывать трейс, теряя детерминизм |
| HTTP API требует ручного управления точками останова | Агент может ошибиться в последовательности вызовов |
| Нет единого контракта «точка → метрики» | Каждый сценарий анализа требует сложной цепочки вызовов |

---

## 3. Бизнес-цель

Реализовать метод `breakpoint-metrics` CLI-стока — детерминированный оркестратор
сбора показателей состояния программы в заданной точке останова.

Метод `breakpoint-metrics`:

- Принимает один JSON-контракт: проект, точка останова (файл + строка), опционально — список переменных для сбора.
- Запускает отладочную сессию, доводит выполнение до точки останова.
- Собирает срез состояния: значения переменных, стек вызовов, контекст потока.
- Возвращает срез в `stdout` в виде JSON.
- Сигнализирует об исходе через детерминированный exit code (0–4).

HTTP-сервер при этом **не изменяется**.

---

## 4. Заинтересованные стороны

| Роль | Интерес |
|------|---------|
| ИИ-агент (LLM-оркестратор) | Получает точный срез состояния без постобработки трейса |
| Разработчик плагина | Реализует эндпоинт `/sink/breakpoint` без изменения ядра |
| QA-инженер | Использует метод для автоматизированных проверок значений переменных |
| Пользователь CLI | Быстро инспектирует состояние в нужной строке кода |

---

## 5. Бизнес-требования

**BR-1.** Метод `breakpoint-metrics` реализует ровно один сценарий: достичь
заданной точки останова и вернуть срез состояния. Никакой другой логики нет.

**BR-2.** Входной контракт и exit-коды стабильны: не меняются без мажорной
версии CLI.

**BR-3.** Метод не содержит бизнес-логики. Вся логика управления точкой
останова и сбора переменных остаётся в серверном компоненте
(`BreakpointMetricsSink.java`). CLI — только валидация, один HTTP-вызов
и маппинг ответа в exit code.

**BR-4.** При любой ошибке агент немедленно знает тип проблемы по exit code
без парсинга тела ответа.

**BR-5.** Метод документирует себя через `--help`.

---

## 6. Ограничения и допущения

- HTTP-сервер `TestActivator` запущен до вызова CLI.
- Точка останова задаётся парой (путь к файлу, номер строки).
- Если точка останова не достигнута за `timeoutMs` — возвращается ошибка exit 3.
- CLI не управляет жизненным циклом HTTP-сервера.

---

## 7. Критерии успеха

- Агент вызывает CLI одной командой и получает exit 0 + JSON со срезом состояния.
- При недостижении точки останова за таймаут — exit 3 с понятным сообщением.
- При неверном JSON на входе — немедленно exit 2, без обращения к HTTP.
- При недоступном HTTP-сервере — exit 1.
- Ни одна строка логики сбора переменных не дублируется в CLI.

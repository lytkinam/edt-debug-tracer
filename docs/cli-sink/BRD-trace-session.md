# BRD — CLI-Сток: метод `trace-session`

**Версия:** 1.1  
**Дата:** 2026-06-06  
**Репозиторий:** lytkinam/edt-debug-tracer, ветка `feature/trace-sink`  
**Статус:** Draft

---

## 1. Бизнес-контекст

Плагин EDT Debug Tracer разворачивает HTTP-сервер внутри Eclipse EDT и предоставляет
программный доступ к отладочной сессии 1C:Enterprise. Инструмент намеренно широк
и расширяем: HTTP API растёт независимо и не ограничивается.

Для встраивания трейсера в конвейеры автономных ИИ-агентов широкий HTTP API
(ручное управление состоянием: `/mcp/start` → `/mcp/run` → `/mcp/stop`)
**не подходит**: агент может зависнуть в промежуточном состоянии, получить
текстовую ошибку и начать галлюцинировать.

**CLI-сток** — детерминированный оркестратор конкретных сценариев использования
EDT Debug Tracer. Каждый сценарий реализован отдельным методом CLI.

Настоящий документ описывает **метод `trace-session`** — полный цикл отладочного
трейса: запуск сессии, пошаговое выполнение, сбор шагов, возврат результата.

---

## 2. Бизнес-проблема

| Проблема | Следствие |
|----------|-----------|
| Агент вызывает `/mcp/start`, затем `/mcp/run` — между вызовами теряется состояние | Трейс начинается без запущенной сессии → ошибка в непредсказуемой точке |
| HTTP возвращает произвольный JSON с текстом ошибки | Агент пытается распарсить текст → галлюцинации вместо retry |
| Нет единого контракта «что подать → что получить» | Каждый сценарий требует от агента знания деталей API |
| Нет явного сигнала «завершено / ошибка» без разбора тела ответа | Агент не знает об успехе без парсинга JSON |

---

## 3. Бизнес-цель

Реализовать метод `trace-session` CLI-стока — детерминированный оркестратор
полного цикла трейса для ИИ-агентов и CI-пайплайнов.

Метод `trace-session`:

- Принимает один JSON-контракт через `stdin` или аргумент.
- Валидирует контракт **до** обращения к HTTP.
- Выполняет ровно один сценарий: `POST /sink/run` → полный цикл трейса.
- Возвращает результат в `stdout` в виде JSON.
- Сигнализирует об исходе через **детерминированный exit code** (0–4).

HTTP-сервер при этом **не изменяется** и продолжает расти независимо.

---

## 4. Заинтересованные стороны

| Роль | Интерес |
|------|---------|
| ИИ-агент (LLM-оркестратор) | Вызывает CLI как чёрный ящик; ожидает exit code и JSON в stdout |
| Разработчик плагина | Сохраняет полный HTTP API без изменений |
| QA / CI-пайплайн | Использует CLI для автоматизированного сбора трейсов |
| Пользователь командной строки | Вызывает CLI вручную для разовых сценариев |

---

## 5. Бизнес-требования

**BR-1.** Метод `trace-session` реализует ровно один сценарий: запуск отладочной
сессии, сбор трейса и возврат результата. Никакой другой логики нет.

**BR-2.** Интерфейс метода стабилен: входной контракт и exit-коды не меняются
без мажорной версии CLI.

**BR-3.** Метод не содержит бизнес-логики. Вся логика трейса остаётся в
`TracerSink.java` / `TestActivator.java`. CLI — только валидация, один
HTTP-вызов и маппинг ответа в exit code.

**BR-4.** При любой ошибке агент немедленно знает **тип** проблемы по exit code
без парсинга тела ответа.

**BR-5.** Метод документирует себя: `--help` возвращает полное описание контракта.

---

## 6. Ограничения и допущения

- HTTP-сервер `TestActivator` запущен до вызова CLI.
- CLI не управляет жизненным циклом HTTP-сервера.
- CLI — Python 3.8+ без внешних зависимостей.

---

## 7. Критерии успеха

- Агент вызывает CLI одной командой и получает exit 0 + JSON с трейсом.
- При неверном JSON на входе — немедленно exit 2, без обращения к HTTP.
- При недоступном HTTP-сервере — exit 1.
- `--help` возвращает полное описание контракта.
- Ни одна строка бизнес-логики трейса не дублируется в CLI.

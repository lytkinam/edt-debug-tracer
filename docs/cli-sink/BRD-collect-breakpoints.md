# BRD — CLI-Сток: метод `collect-breakpoints`

**Версия:** 1.0  
**Дата:** 2026-06-06  
**Репозиторий:** lytkinam/edt-debug-tracer, ветка `feature/trace-sink`  
**Статус:** Draft

---

## 1. Бизнес-контекст

CLI-сток — детерминированный оркестратор конкретных сценариев использования
EDT Debug Tracer. Каждый метод — отдельный сценарий с неизменяемым контрактом.

Метод `breakpoint-metrics` (см. BRD-breakpoint-metrics.md) нацелен на
**одну** точку останова: доходит до неё, снимает срез, возвращает ответ.  
Метод `trace-session` (см. BRD-trace-session.md) пишет полный трейс шагов.

Оба метода не решают сценарий: **«собрать срезы состояния по нескольким точкам
за один прогон программы»** — наиболее частый случай при анализе поведения
1C-кода агентом: проверить, какие значения были в точке A, B, C при единственном
запуске, не запуская программу три раза.

---

## 2. Бизнес-проблема

| Проблема | Следствие |
|----------|-----------|
| `breakpoint-metrics` обслуживает одну точку за запуск | Анализ N точек требует N запусков, N отдельных EDT-сессий |
| Режим polling (`poll_break_status`) зависит от цикла ожидания в CLI | Если поток засыпает, механизм `sleep` пропускает быстрые остановки |
| Внутри Eclipse уже есть `ISuspendListener` — событийный перехват остановки потока | Этот механизм не используется: потенциал Eclipse DSP-модели игнорируется |
| Нет способа собрать точки «в фоне», пока программа идёт | Агент не может сказать «пусти программу и скажи, где она останавливалась» |

---

## 3. Бизнес-цель

Реализовать режим `collect_breakpoints` для HTTP-стока и соответствующий
метод `collect-breakpoints` CLI-стока.

**Режим `collect_breakpoints`:**

- Вводится флаг `collect_breakpoints: true` в методах `start` и `resume`.
- При активном флаге система **не ждёт** команды от оркестратора после каждой
  остановки — вместо этого **автоматически возобновляет** выполнение и
  **накапливает снимки** (файл, строка, переменные, стек) по каждому срабатыванию.
- Движение по коду управляется исключительно внутренним механизмом Eclipse —
  `ISuspendListener` — без `Thread.sleep` и polling-циклов в CLI.
- По завершении программы (или по таймауту) результаты доступны через
  `get_collect_breakpoints(session_id)`.

**Метод `collect-breakpoints` CLI-стока:**

- Принимает JSON: проект, список точек останова (файл + строка + опциональный
  фильтр переменных), параметры запуска, таймаут.
- Автоматически снимает все ранее установленные точки, устанавливает только свои.
- Вызывает `start` с `collect_breakpoints: true`.
- Ожидает завершения через `poll_session_status` (не `poll_break_status`).
- Вызывает `get_collect_breakpoints(session_id)` для сбора результатов.
- Снимает установленные точки.
- Возвращает `stdout` JSON + детерминированный exit code.

## 4. Обоснование реализации на стороне Eclipse (PR-аргументация)

> **Ключевой принцип:** движение по коду не должно зависеть от функции ожидания
> (polling, `Thread.sleep`) в CLI или сервере.

Внутри Eclipse Platform Debug есть событийный контракт:

```
org.eclipse.debug.core.IDebugEventSetListener
   └── слушает DebugEvent.SUSPEND
           └── поле detail: BREAKPOINT | STEP_END | CLIENT_REQUEST | ...
```

и для DSP-отладки:

```
ISuspendListener.suspended(IStackFrame frame, IThread thread)
```

Когда поток останавливается на точке останова, Eclipse уведомляет всех
зарегистрированных слушателей **синхронно, в том же потоке отладки**, до любого
UI-взаимодействия. Это означает:

| Подход | Риск пропустить событие | Задержка накопления | Зависимость от таймера |
|--------|------------------------|--------------------|-----------------------|
| `Thread.sleep` + polling REST | Да (при sleep > времени остановки) | Высокая | Да |
| `ISuspendListener` (Eclipse events) | Нет | Нулевая | Нет |

**Вывод:** реализация через `ISuspendListener` исключает гонку состояний,
гарантирует захват каждой остановки и не требует цикла ожидания ни в CLI,
ни в серверном обработчике.

Реализация на стороне HTTP-стока:

```java
// При start/resume с collect_breakpoints=true:
IDebugTarget target = session.getDebugTarget();
target.getDebugTarget().addDebugEventListener(new CollectBreakpointsListener(session));
// ...
class CollectBreakpointsListener implements IDebugEventSetListener {
    public void handleDebugEvents(DebugEvent[] events) {
        for (DebugEvent e : events) {
            if (e.getKind() == DebugEvent.SUSPEND
                    && e.getDetail() == DebugEvent.BREAKPOINT) {
                collector.capture(session, (IThread) e.getSource());
                ((IThread) e.getSource()).resume(); // автовозобновление
            }
        }
    }
}
```

## 5. Заинтересованные стороны

| Роль | Интерес |
|------|---------|
| ИИ-агент | Получает срезы по всем точкам за один запрос, без N запусков |
| Разработчик плагина | Реализует `ISuspendListener` вместо polling — меньше кода, нет гонок |
| QA-инженер | Автоматически верифицирует несколько инвариантов за прогон |
| Пользователь CLI | Одна команда → все точки → полный ответ |

## 6. Бизнес-требования

**BR-1.** Флаг `collect_breakpoints: true` в `start` / `resume` активирует
режим автоматического накопления снимков по точкам останова.

**BR-2.** При `collect_breakpoints: true` HTTP-сток **никогда** не ждёт команды
от внешнего клиента между остановками — продолжение выполнения инициируется
через Eclipse `IDebugEventSetListener`, не через polling.

**BR-3.** Метод `get_collect_breakpoints(session_id)` возвращает список
накопленных снимков в любой момент и по завершении сессии.

**BR-4.** Оркестратор CLI (`collect-breakpoints`) управляет жизненным циклом
точек: снимает чужие, устанавливает свои, возвращает к исходному состоянию.

**BR-5.** Контракт CLI неизменяем без мажорной версии.

**BR-6.** При любой ошибке агент знает тип проблемы по exit code без парсинга тела.

## 7. Ограничения и допущения

- HTTP-сервер `TestActivator` запущен до вызова CLI.
- Повторный вызов `set_breakpoint` на той же позиции снимает точку (toggle).
- Оркестратор не восстанавливает точки других агентов: он снимает все, устанавливает
  свои, а после сбора — снимает только свои.
- `get_collect_breakpoints` доступен только пока сессия жива или завершилась
  нормально; при `INTERNAL_ERROR` данные могут быть частичными.

---

## 8. Критерии успеха

- Агент вызывает `collect-breakpoints` с N точками — получает exit 0 + JSON
  со снимками по каждой сработавшей точке за один прогон.
- При незасработавшей точке — поле `hits: []` для этой точки, exit 0.
- При любой точке без `hit` за `timeoutMs` — exit 3.
- Ни один `Thread.sleep` / polling не используется в Eclipse-части реализации.
- Цикл ожидания в CLI — только `poll_session_status` (не `poll_break_status`).

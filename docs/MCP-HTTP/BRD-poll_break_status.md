# BRD — HTTP-метод: `poll_break_status`

**Версия:** 1.0  
**Дата:** 2026-06-06  
**Статус:** Draft — запрос от оркестратора (замена `wait_for_break`)  
**Владелец контракта:** CLI-оркестратор (`tracer-sink`)

---

## Потребители контракта

| Компонент | Роль | Как использует |
|---|---|---|
| `tracer-sink` CLI | Оркестратор | Шаг 3 сценария `breakpoint-metrics`: повторяет вызов с интервалом `pollIntervalMs` до получения `suspended: true` или истечения `timeoutMs` |

---

## Почему не `wait_for_break`

`wait_for_break` — блокирующий HTTP-вызов: соединение держится открытым до события SUSPEND. Это создаёт критическую проблему:

| Проблема | Следствие |
|---|---|
| Обрыв соединения на шаге 3 | Оркестратор получает `exit 1`, но отладчик уже остановлен — состояние рассинхронизировано |
| Невозможно повторить вызов идемпотентно | Повторный `wait_for_break` может пропустить уже произошедшее событие SUSPEND |
| HTTP-таймауты инфраструктуры | Прокси, балансировщики, firewall обрывают соединения длиннее 30–60 секунд |

---

## Бизнес-цель

Метод `poll_break_status` — **неблокирующий**: возвращает немедленно с текущим статусом потока. Оркестратор сам управляет циклом опроса: вызывает метод с интервалом, сам контролирует таймаут, сам восстанавливается после разрыва сети (повторяет опрос).

Преимущества:
- **Идемпотентен**: повторный вызов после разрыва не теряет событие SUSPEND — статус хранится в HTTP-стоке
- **Управляемый таймаут**: оркестратор считает время сам, не зависит от HTTP-инфраструктуры
- **Наблюдаемость**: оркестратор видит промежуточные статусы (`running`, `stepping`) и может логировать прогресс

---

## Критерии успеха

- Метод возвращает текущий статус потока немедленно (без ожидания события).
- При `suspended: true` возвращает `threadId`, `file`, `line`.
- Оркестратор за `timeoutMs` получает либо `suspended: true`, либо завершает с `exit 3`.
- Повторный вызов после сетевого разрыва возвращает корректный статус.

# SRS — HTTP-метод: `get_capabilities`

**Версия:** 1.0  
**Дата:** 2026-06-06  
**Статус:** Draft — ожидает реализации HTTP-стоком

---

## Потребители контракта

| Компонент | Роль | Как использует |
|---|---|---|
| `tracer-sink` CLI | Оркестратор | Шаг 0: `GET /mcp/capabilities` → проверить список `methods[]` |

---

## Эндпоинт

```
GET http://{host}:{port}/mcp/capabilities
```

> Это единственный метод, использующий `GET`, а не `POST /mcp`. Обоснование: не изменяет состояние, не требует тела запроса, кешируемый.

---

## Запрос

Тело запроса: отсутствует.

---

## Ответ (HTTP 200)

```json
{
  "methods": [
    "resume",
    "get_call_stack",
    "get_variables",
    "suspend"
  ],
  "version": "1.0"
}
```

| Поле | Тип | Обязательное | Описание |
|---|---|:---:|---|
| `methods` | `string[]` | ✅ | Список имён реализованных методов |
| `version` | `string` | ✅ | Версия HTTP-стока |

---

## Поведение при ошибках

| HTTP-код | Условие | Действие оркестратора |
|---|---|---|
| Нет соединения | HTTP-сервер недоступен | `exit 1` |
| 200 | `methods[]` не содержит нужный метод | `exit 3` с сообщением: `«Метод X не реализован»` |

---

## Логика оркестратора

```
1. GET /mcp/capabilities
2. Если соединение не установлено → exit 1
3. Для каждого required_method сценария:
   если method НЕ в methods[] → exit 3 «{method} не реализован»
4. Все методы присутствуют → запустить сценарий
```

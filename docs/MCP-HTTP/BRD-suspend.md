# BRD — HTTP-метод `suspend`

**Версия:** 1.0  
**Дата:** 2026-06-06  
**Статус:** ✅ Метод существует  
**Связанный SRS:** [SRS-suspend.md](SRS-suspend.md)

---

## 1. Контекст

После сбора стека и переменных оркестратор завершает сценарий.
Поток остаётся в состоянии SUSPEND (программа не продолжает выполнение).
`suspend` — финальная фиксация состояния: оркестратор явно сигнализирует
отладчику, что сбор завершён и поток можно считать управляемым.

---

## 2. Потребность оркестратора

Оркестратор вызывает `suspend` **последним** в сценарии, чтобы:
1. Убедиться, что поток не продолжил выполнение самостоятельно.
2. Оставить отладочную сессию в предсказуемом состоянии для следующих вызовов.

Важно: **не `resume`** — оркестратор не должен продолжать программу,
когда его цель — зафиксировать состояние в точке.

---

## 3. Место в сценарии

```
[ШАГ 5] get_variables
           │
           ▼
[ШАГ 6] suspend(project, threadId)   ← этот метод
           │
           ▼
        stdout JSON, exit 0
```

---

## 4. Ожидаемое поведение

1. Если поток уже в SUSPEND — идемпотентен, не возвращает ошибку.
2. Возвращает управление синхронно.

# BRD — HTTP-метод: `get_capabilities`

**Версия:** 1.0  
**Дата:** 2026-06-06  
**Статус:** Draft — запрос от оркестратора  
**Владелец контракта:** CLI-оркестратор (`tracer-sink`)

---

## Потребители контракта

| Компонент | Роль | Как использует |
|---|---|---|
| `tracer-sink` CLI | Оркестратор | Шаг 0 каждого сценария: проверяет доступность нужных методов перед запуском |

---

## Бизнес-контекст

Оркестратор реализует несколько сценариев (`trace-session`, `breakpoint-metrics`), каждый из которых требует разного набора HTTP-методов. Без информации о доступных методах оркестратор вынужден запускать сценарий и обнаруживать отсутствие метода только на конкретном шаге — через `exit 1/3`.

Это создаёт проблему: сценарий может частично выполниться (например, установить точку останова), прежде чем обнаружит, что следующий метод отсутствует. Откат такого состояния нетривиален.

---

## Бизнес-проблема

| Проблема | Следствие |
|---|---|
| Оркестратор не знает заранее, какие методы реализованы | Частичное выполнение сценария при отсутствии метода на середине |
| Агент вынужден интерпретировать `exit 1/3` чтобы понять причину | Неоднозначность: сеть упала или метод не реализован? |

---

## Бизнес-цель

Метод `get_capabilities` — единственный вызов оркестратора **до начала сценария**. Возвращает список реализованных методов HTTP-стока. Оркестратор проверяет наличие всех нужных методов и либо запускает сценарий, либо немедленно возвращает `exit 3` с понятным сообщением: `«set_breakpoint не реализован»`.

---

## Критерии успеха

- Оркестратор получает список методов за один вызов до старта сценария.
- При отсутствии нужного метода — `exit 3` с именем отсутствующего метода, без запуска сценария.
- Метод не требует аргументов и не изменяет состояние отладчика.

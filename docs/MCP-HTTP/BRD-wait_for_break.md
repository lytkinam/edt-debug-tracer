# BRD — HTTP-метод `wait_for_break`

**Версия:** 1.0  
**Дата:** 2026-06-06  
**Статус:** ✅ Метод существует  
**Связанный SRS:** [SRS-wait_for_break.md](SRS-wait_for_break.md)

---

## 1. Контекст

После `resume` программа выполняется асинхронно. Оркестратор не знает,
когда именно программа достигнет точки останова. Ему нужен
**блокирующий ожидатель события** — метод, который удерживает HTTP-соединение
открытым до наступления SUSPEND или до истечения таймаута.

`wait_for_break` — единственный способ, которым оркестратор
детерминированно узнаёт о факте достижения точки.

---

## 2. Потребность оркестратора

Оркестратор вызывает `wait_for_break` сразу после `resume` и блокируется
на время до `timeoutMs` мс. Из ответа он получает `threadId` — он нужен
для последующих вызовов `get_call_stack` и `get_variables`.

Если `timeout: true` — программа не достигла точки за отведённое время.
Оркестратор завершает сценарий с `exit 3`.

---

## 3. Место в сценарии `breakpoint-metrics`

```
[ШАГ 2] resume
           │
           ▼
[ШАГ 3] wait_for_break(project, timeoutMs)   ← этот метод
           ├── timeout: true  →  exit 3
           └── suspended: true
                   │
                   ▼
           [ШАГ 4] get_call_stack
```

---

## 4. Ожидаемое поведение

1. Метод **блокирующий** — HTTP-ответ приходит только после события или таймаута.
2. Таймаут управляется параметром `timeoutMs` от оркестратора.
3. В ответе обязан быть `threadId` при `suspended: true` — без него оркестратор
   не сможет собрать стек и переменные.
